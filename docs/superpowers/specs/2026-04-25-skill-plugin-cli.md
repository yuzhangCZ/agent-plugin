# `skill-plugin-cli` 需求说明

日期：2026-04-25

## 背景与问题陈述

当前 `message-bridge` / `message-bridge-openclaw` 的安装能力分别散落在各自插件包内。

已知问题如下：

1. OpenClaw 在 `2026.4.1` 及以上版本中，`openclaw plugins install` 安装 `message-bridge-openclaw` 的链路存在现实阻断；该阻断发生在安装期安全限制场景下。
2. `message-bridge` / `message-bridge-openclaw` 两端安装脚本都需要支持二维码认证，但公共逻辑分散维护两套，执行流程缺乏统一，长期维护成本较高。
3. 当前安装能力除二维码认证外，还包含宿主环境校验、插件安装、安装结果校验、宿主配置接入等流程，尚未形成统一需求口径。

为解决上述问题，需要补充一份面向统一安装场景的需求说明，作为后续能力收敛的需求依据。

## 目标

本需求的目标是定义一套面向 `cli install` 场景的统一安装能力边界，并以 `packages/skill-plugin-cli` 作为该能力的统一承载方向。

本期目标包括：

1. 明确 `skill-plugin-cli` 在统一安装场景中的职责边界。
2. 统一 OpenCode / OpenClaw 双端安装需求口径。
3. 将二维码认证、插件安装、安装校验、宿主配置接入纳入同一需求对象中描述。
4. 为 OpenClaw 新版本安装阻断场景提供独立 CLI 承接路径的明确需求承接。

## 非目标

本期不覆盖以下内容：

- `skill-plugin-cli` 的内部模块拆分与实现架构。
- 对外 CLI 命令、参数表、目录结构或发布形态设计。
- 二维码 ASCII 渲染方案、第三方库选型、终端重绘机制。
- 配置文件路径、字段 schema、宿主命令编排细节。
- 历史版本迁移的具体执行方案与兼容实现。
- 对 `skill-qrcode-auth`、`message-bridge`、`message-bridge-openclaw` 的逐文件改造步骤。

## 本期范围

本期需求范围聚焦于 `cli install` 场景，并同时覆盖：

- OpenCode
- OpenClaw

本需求关注的是统一安装能力边界，而不是保留两套彼此独立演进的安装逻辑。

## 用户可见结果

特性完成后，用户在安装流程中应能看到以下结果：

1. 用户能够在安装过程中看到可扫码二维码，并使用扫码方式完成认证，而不再仅依赖手动输入凭据。
2. 终端展示中使用 `weUrl` 作为二维码扫码内容，并同时展示 `pcUrl` 作为辅助信息。
3. 安装流程可持续感知二维码生成、扫码、过期、取消、确认、失败等认证结果。
4. 二维码过期后，用户能够继续进入后续授权流程。
5. 授权成功后，安装流程能够获得标准结果 `ak/sk`，并完成后续宿主侧接入。
6. 安装完成后，用户能够确认宿主侧接入结果已经可用。
7. 在支持终端超链接能力的环境中，用户可点击 `weUrl` 并通过浏览器打开；在不支持该能力的环境中，用户仍可看到完整 `weUrl` 文本并继续授权。

二维码展示的用户可见效果可参考现有同类 CLI 产品的终端交互形态，例如 `@larksuite/openclaw-lark`、`@tencent-weixin/openclaw-weixin-cli`；本文档不将这些参考对象定义为实现依赖。

### 终端二维码展示要求

本需求在用户可见层面对终端二维码展示提出以下要求：

1. `weUrl` 是终端中的主扫码入口，`pcUrl` 是辅助信息，不能以仅展示 `pcUrl` 替代 `weUrl` 展示。
2. 终端展示可以是二维码图形或其他可扫码等效展示；本文档不规定具体渲染方案。
3. 在支持终端超链接能力的环境中，`weUrl` 应以可点击方式展示，用户可通过浏览器打开。
4. 在不支持终端超链接能力，或二维码图形无法正常展示的环境中，安装流程至少应向用户展示完整 `weUrl` 文本，使用户仍可通过复制或浏览器打开继续授权。
5. 当前二维码过期时，终端应明确告知该二维码已失效或不可继续使用。
6. 若授权流程允许继续，终端应继续向用户提供新的可用扫码入口，并使用户能够明确区分“旧二维码已失效”与“新二维码已可继续扫码”。

## 核心能力边界

`skill-plugin-cli` 作为统一安装能力承载体，在需求层至少应覆盖以下能力：

1. 宿主环境校验。
2. 插件安装。
3. 插件安装校验。
4. 二维码认证。
5. 认证结果接入与宿主可用配置接入。
6. 安装所需上下文输入承接。
7. `@wecode` 二方仓源配置。

上述能力含义如下：

- “插件安装校验”是独立能力，不等同于安装前 preflight；其目标是确认插件已进入宿主可识别、可继续配置或可使用状态。
- “认证结果接入与宿主可用配置接入”表示安装流程在认证成功后，不仅要获得 `ak/sk`，还要完成宿主侧所需接入动作，使插件、通道或等效能力进入宿主可识别、可继续配置或可继续使用状态。该能力不以单纯获得或写入凭据为完成条件。
- “安装所需上下文输入承接”指统一安装能力需要接收宿主安装流程所需的必要上下文，而不是只抽象描述为“安装目标”。
- “`@wecode` 二方仓源配置”属于统一安装能力的一部分，不应被视为无关实现细节。

## 宿主范围与约束

本需求对宿主侧约束明确如下：

1. OpenCode / OpenClaw 双端都在统一安装需求范围内。
2. 公共安装能力应收敛到统一承载体中，不能继续以两套彼此独立维护的安装逻辑作为长期目标形态。
3. 不同宿主可以保留必要差异，但这些差异应建立在统一安装能力边界之上。
4. 统一安装能力需要支持宿主环境校验，但不同宿主可因能力模型不同而具有不同校验语义和阻断级别。
5. 安装完成不应仅以执行安装命令或获得认证结果为准，还应支持安装后的生效或结果确认。
6. 当 OpenClaw 宿主原生 `openclaw plugins install` 链路因安装期安全限制不可用时，统一安装能力仍应支持通过独立 CLI 完成安装流程。

## 事实输入与参考对象

以下内容属于当前需求依赖的事实输入或参考对象，用于说明需求来源与用户可见预期，不代表本期已确定具体实现方案。

### 1. OpenClaw 现实阻断

- OpenClaw `2026.4.1+` 的安装链路存在现实阻断。
- 阻断发生在 `openclaw plugins install` 安装场景中。
- 阻断与宿主对安装期能力的安全限制有关。
- 该阻断场景可通过 CLI 单独拆包承接，不应留给实现阶段自行解释。

### 2. 双端现状

- OpenCode 与 OpenClaw 当前都存在安装期能力，但能力边界、执行流程与公共逻辑尚未统一描述。
- 两端现有安装流程都不仅包含二维码认证，还包含宿主环境校验、插件安装、安装结果校验、宿主配置接入等能力。

### 3. 二维码展示参考对象

- `@larksuite/openclaw-lark`
- `@tencent-weixin/openclaw-weixin-cli`

上述对象仅作为用户可见交互参考，不在本需求中定义为实现依赖、技术选型或兼容承诺。

## 验收标准

补充后的统一安装需求至少应满足以下验收口径：

1. 双端安装场景具备统一需求描述，不再分散为两套彼此独立的需求口径。
2. 需求中明确覆盖宿主环境校验、插件安装、插件安装校验、二维码认证、认证结果接入等安装期核心能力。
3. “认证结果接入与宿主可用配置接入”被保留为明确完成条件，不被弱化为泛泛的“拿到并使用凭据”。
4. “插件安装校验”被明确写成独立能力和独立验收项。
5. “认证结果接入与宿主可用配置接入”覆盖到使插件或通道进入可用状态的宿主侧接入动作，而不只停留在 `ak/sk` 获取层。
6. “`@wecode` 二方仓源配置”被纳入统一安装能力边界。
7. 安装完成条件中包含安装后的生效或结果确认要求。
8. 二维码部分明确 `weUrl`、`pcUrl`、授权状态变化、过期继续授权、成功结果 `ak/sk` 等用户可见要求。
9. 在支持终端超链接的环境中，`weUrl` 可点击并可通过浏览器打开；在不支持超链接或二维码图形无法展示的环境中，终端仍会展示完整 `weUrl` 文本。
10. 当前二维码过期表示该二维码实例已不可继续使用，但不必然表示整次授权流程立即失败；若流程仍允许继续，终端会继续提供新的可用扫码入口；若流程不能继续，则以明确失败或结束结果收口。
11. OpenClaw `2026.4.1+` 宿主原生安装链路受阻时，统一安装需求仍定义了可完成安装流程的独立 CLI 承接路径。
12. 文档不包含模块设计、命令参数、命令编排、终端渲染实现、配置字段 schema 等方案内容。

## 结论

本需求将 `skill-plugin-cli` 定义为统一安装场景的能力承载方向，用于统一描述 OpenCode / OpenClaw 双端在 `cli install` 中涉及的宿主环境校验、插件安装、插件安装校验、二维码认证、认证结果接入与宿主可用配置接入等能力边界。

本期重点是补齐需求描述并统一口径，而不是提前确定实现方案、内部结构或具体 CLI 形态。
