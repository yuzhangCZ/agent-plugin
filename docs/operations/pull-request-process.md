# Pull Request 流程规范

## 目标

统一本仓库 PR 提交流程，减少“口头规范”与“执行结果”不一致的问题。

## 适用范围

- 仓库根目录及 `plugins/`、`packages/` 下的日常开发 PR
- 不覆盖子目录更严格的专用规范（如 `plugins/message-bridge/docs/AGENTS.md`）

## 单一事实源

- 入口强约束：`AGENTS.md` 的“PR 入口规则”
- 详细描述模板：`.github/PULL_REQUEST_TEMPLATE.md`
- 本文档：流程步骤、预检清单、冲突处理方式

## 标准流程

1. 从 `main` 切出功能分支（建议 `codex/<topic>`）。
2. 仅暂存本次改动，避免混入无关文件。
3. 本地执行受影响验证命令（至少受影响包测试）。
4. 提交 commit（信息需明确范围与目的）。
5. 推送分支并创建 PR，目标分支默认 `main`。
6. 按模板填写，并逐项核对 PR 描述、行为变化和验证证据与实际改动一致。

## 规范预检清单（创建 PR 前）

- [ ] 当前变更范围符合仓库边界（默认仅 `plugins/`、`packages/`）
- [ ] 已执行受影响验证命令，结果可复现
- [ ] PR 目标分支已确认（默认 `main`）
- [ ] PR 描述使用模板结构，包含测试结果与风险说明
- [ ] PR 描述已写明当前行为、改后行为、变化点和未变化边界
- [ ] 主要行为变化已逐项对应到验证证据
- [ ] 默认值变化 / 兼容性变化 / 迁移影响已显式声明
- [ ] 如存在规范冲突，已先确认再执行

## 冲突停机规则（必须）

出现以下任一情况时，停止自动执行并先确认：

- 通用流程要求与仓库事实冲突（例如建议的 base 分支不存在）
- 找不到模板或模板与现有流程不一致
- 改动涉及默认行为变化，但 PR 未清楚说明影响范围和回滚方式
- 需要在“速度优先”和“规范完整”之间做取舍

## 说明

- 本文档用于避免重复决策，不替代 CI 校验。
- 若后续新增自动化门禁，应以本文档为准更新脚本行为。
