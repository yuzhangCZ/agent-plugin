# OpenCode 实战指南：初学者如何真正用起来

> 本文基于 2026 年 3 月 17 日检索到的 OpenCode 官方中文文档、官网首页与 GitHub 仓库资料整理而成。安装方式、命令、权限模型与扩展能力以官方文档为准；其中 `session`、`context` 的部分表述，属于面向使用者的归纳说明。

## 如果你刚开始接触 OpenCode，这篇文章想解决什么

如果你刚开始接触 AI coding agent，这份指南主要帮你解决两件事：

- 第一次怎么真正开始用 `OpenCode`
- `skill`、`custom tools`、`MCP`、`plugin` 到底怎么区分

如果你时间不多，先记住这几句：

- 对初学者，最先要掌握的不是 `plugin`
- 你真正每天会用到的是 `session`、`agent`、`permission`、`context`、`rules`
- 真正开始进阶时，通常是先学会 `skill`
- 需要给 AI 增加外部工具能力时，再看 `custom tools` 或 `MCP`
- 只有当你要扩展 OpenCode 自身运行时，才需要考虑 `plugin`

## 1. 先把 OpenCode 想对：它是开发工作台，不只是聊天窗口

`OpenCode` 是一个开源 AI coding agent。它既可以运行在 Terminal，也可以配合 Desktop 和 IDE 使用。它支持连接不同 provider，也支持通过自定义规则、技能、工具和扩展能力，逐步把“AI 参与开发”这件事做成稳定流程。

从初学者视角，你可以先把它理解成一套开发工作台，而不是一个聊天窗口：

- 它能理解你的项目规则
- 它能读取文件、执行命令、修改代码
- 它能把一个需求拆成分析、设计、实现、验证几个阶段
- 它能把经验沉淀成 `AGENTS.md`、`instructions`、`skill`
- 它还能接入外部工具，甚至扩展自身运行时

真正重要的不是“它会不会写代码”，而是你能不能让它在正确的上下文里，按正确的权限和流程工作。后面整篇文章，其实都在解决这一件事。

## 2. 先别研究概念，10 分钟把第一次使用跑通

这一节的目标不是讲概念，而是让你尽快跑通第一次上手。

### 2.1 准备环境

- macOS / Linux 可以直接使用
- Windows 官方建议优先通过 `WSL` 使用
- 你需要准备一种可用的模型接入方式

### 2.2 安装 OpenCode

最常见的安装方式是安装脚本：

```bash
curl -fsSL https://opencode.ai/install | bash
```

如果你更习惯 Node.js 工具链，也可以使用：

```bash
npm install -g opencode-ai
```

```bash
pnpm add -g opencode-ai
```

### 2.3 进入项目并启动

先进入你要处理的项目目录，再启动 `OpenCode`：

```bash
cd /path/to/project
opencode
```

### 2.4 第一次连接 provider

进入 TUI 后，先运行：

```text
/connect
```

官方文档建议初学者可以先选择 `opencode` 并前往 `opencode.ai/auth` 完成登录和 API Key 配置。完成后，你就有了可用模型。

### 2.5 初始化项目规则

第一次在某个仓库里使用时，建议立刻运行：

```text
/init
```

`OpenCode` 会分析项目并在仓库根目录创建 `AGENTS.md`。官方也明确建议把项目级 `AGENTS.md` 提交到 Git，因为这会成为团队共享的项目规则入口。

### 2.6 先用 Plan 做分析

`OpenCode` 内置 `Build` 和 `Plan` 两个主代理。对初学者，推荐先切到 `Plan`，再开始第一轮需求分析。

- 使用 `Tab` 可以在主代理之间切换
- `Plan` 适合分析、审查建议、做方案
- `Build` 适合真正开始实现

第一段最小提示模板：

```text
请先不要写代码。
基于当前仓库，帮我分析这个需求：
[在这里写你的需求]

输出请包含：
1. 目标和边界
2. 可能受影响的文件
3. 技术风险和不确定点
4. 推荐的最小改动方案
5. 需要补充的测试和验证项
```

### 2.7 再切到 Build 做实现

当 `Plan` 已经把边界和方案讲清楚后，再切到 `Build`，开始实施。

第二段最小提示模板：

```text
现在基于上面的分析开始实现。
要求：
1. 先修改最小必要代码
2. 再补测试或更新测试
3. 运行可以运行的验证命令
4. 按下面格式输出：
   - 实际修改了什么
   - 运行了什么验证
   - 哪些通过
   - 哪些没跑
   - 剩余风险是什么
```

### 2.8 如果方向错了，及时撤销

如果实现结果不对，可以使用：

```text
/undo
```

如果你撤销后又想恢复，可以使用：

```text
/redo
```

### 2.9 这几个命令先记住

| 命令 | 作用 |
| --- | --- |
| `/connect` | 连接 provider |
| `/init` | 生成项目级 `AGENTS.md` |
| `/undo` | 撤销最近一次修改 |
| `/redo` | 重做刚撤销的修改 |
| `/share` | 生成当前会话分享链接 |
| `/unshare` | 取消分享并删除相关分享数据 |
| `/compact` | 压缩当前会话，适合上下文变长时使用 |

另外两个高频输入方式也很重要：

- 用 `@文件路径` 引用文件，文件内容会被自动加入上下文
- 用 `!命令` 运行 shell 命令，结果会作为工具输出进入对话

## 3. 真正会长期困住新手的，是这些概念的层级

这部分是整篇文档最重要的地方。很多人第一次接触 OpenCode 时，会把 `plugin`、`skill`、`session`、`context` 全都看成“差不多的概念”。这正是最容易把自己绕进去的地方。

更稳妥的做法，是先按层级理解：

- 工作容器：`session`
- 执行角色：`agent`
- 安全控制：`permission`
- 认知输入：`context`、`rules`、`AGENTS.md`、`instructions`
- 工具能力：内置 `tools`、`custom tools`、`MCP`
- 方法与运行时扩展：`skill`、`plugin`

### 3.1 工作容器：`session`

**它是什么**

- `session` 可以理解为一次完整的工作线程，也就是你当前这轮对话、分析、实现和验证所在的会话容器

**为什么你需要关心**

- 它决定了当前任务的历史消息、上下文积累和后续复盘边界
- 子代理还可能产生子会话，因此 `session` 不只是聊天记录，而是工作结构的一部分

**初学者第一次通常怎么用**

- 一个明确需求对应一个 `session`
- 一个 bug、一项小功能、一次文档修改，尽量不要混在同一个 `session` 里

**什么时候先不要做**

- 不要把“修 bug”“重构”“补文档”“查问题”塞进同一条会话
- 如果上下文已经明显跑偏，宁可开一个新 `session`，也不要硬接着聊

### 3.2 执行角色：`agent`

**它是什么**

- `agent` 是不同工作模式下的执行者
- 官方文档把它定义为“可针对特定任务和工作流程配置的专门 AI 助手”

**为什么你需要关心**

- 因为你不是总在做同一件事
- 有时你需要先分析，有时你需要真正实施，有时你需要子代理帮你做只读探索

**初学者第一次通常怎么用**

- `Plan`：先分析需求、看方案、做审查，不急着改代码
- `Build`：在方案明确后开始实施
- 使用 `Tab` 在主代理之间切换
- 使用 `@agent` 调子代理做专项任务

**什么时候先不要做**

- 不要一上来就一直用 `Build`
- 如果你还没讲清楚目标和边界，就先别让它直接改代码

### 3.3 安全控制：`permission`

**它是什么**

- `permission` 决定某个动作是自动执行、需要审批，还是直接禁止
- 官方权限模型的三个基本动作是：`allow`、`ask`、`deny`

**为什么你需要关心**

- 因为 OpenCode 默认是“工具已启用且可直接运行”的思路
- 如果你不主动配置权限，AI 改文件和跑命令的自由度会比很多初学者预期更高

**初学者第一次通常怎么用**

- 初期优先使用更保守的策略，例如：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "*": "ask"
  }
}
```

- 或至少把 `edit`、`bash` 收紧到 `ask`

**什么时候先不要做**

- 不要把“安全使用 OpenCode”理解成只靠提示词约束
- 真正可靠的是权限配置，而不是“请不要乱改文件”这类口头说明

### 3.4 `Plan` 和 `Build` 的关键区别

很多初学者以为 `Plan` 和 `Build` 的区别只是“一个负责想，一个负责做”。这个理解不够完整。

官方文档明确写到：

- `Build` 是默认主代理，并启用了所有工具
- `Plan` 是一个受限代理
- `Plan` 默认会把 `file edits` 和 `bash` 都设置为 `ask`

对使用者来说，真正应该记住的是：

- `Plan` 适合先分析、先收敛方案
- `Build` 适合开始实现
- 安全感来自权限边界，而不是来自代理名字

### 3.5 认知输入：`context`

**它是什么**

- `context` 是当前模型真正“看到了什么”
- 官方没有把它写成单独术语页，但它实际上贯穿文件引用、规则、命令输出、技能加载、会话压缩等能力

**为什么你需要关心**

- 很多时候 AI 不是“不会做”，而是“没看到”
- 上下文质量，通常比提示词花样更重要

**初学者第一次通常怎么用**

- 明确给出目标、范围、约束、验收
- 用 `@` 引用关键文件
- 让它先读 `README`、关键配置和核心入口文件
- 需要命令结果时，用 `!` 显式把输出放进上下文

**什么时候先不要做**

- 不要指望一条非常抽象的话就让它自动理解整个项目
- 不要在一个过长、过杂的会话里继续堆需求

### 3.6 项目规则：`rules`、`AGENTS.md`、`instructions`

**它是什么**

- `AGENTS.md` 是最常见的项目规则入口
- `opencode.json` 中的 `instructions` 也是规则输入
- 官方文档明确说明：`instructions` 中的文件会和 `AGENTS.md` 合并，一起进入模型上下文

**为什么你需要关心**

- 因为这是项目级长期约束
- 它们决定 OpenCode 对“这个仓库该怎么工作”的长期理解

**初学者第一次通常怎么用**

- 先运行 `/init` 生成初始 `AGENTS.md`
- 把目录边界、常用命令、测试方式、代码风格写进去
- 如果团队已经有现成规则文件，优先通过 `instructions` 复用，而不是手工复制进 `AGENTS.md`

**什么时候先不要做**

- 不要把 `rules` 简化成“只有一个 `AGENTS.md` 文件”
- 不要假设 `AGENTS.md` 中引用的外部文件会被自动解析

一个很实用的理解方式是：

- `session` 是任务级短期工作容器
- `rules` 是项目级长期约束
- `context` 是当前这轮模型真正看到的全部输入

### 3.7 工具能力：内置 `tools`

**它是什么**

- 官方文档把 `tools` 定义为“LLM 在代码库中执行操作的能力”
- OpenCode 自带一组内置工具，比如读文件、列目录、执行命令、编辑文件、提问、Web 获取等

**为什么你需要关心**

- 因为 AI 并不是凭空写代码，它是通过工具真正观察和操作项目
- 你看到的很多能力差异，最后都落在工具是否可用、权限是否放开

**初学者第一次通常怎么用**

- 用 `@` 让它读取关键文件
- 让它运行必要的测试和构建命令
- 通过权限决定哪些动作可自动执行

**什么时候先不要做**

- 不要默认所有工具都该一直开启
- 不要忽略权限配置和工具粒度控制

### 3.8 方法复用：`skill`

**它是什么**

- 官方文档把 `skill` 描述为由 `SKILL.md` 定义、可被按需加载的可复用行为
- 它更像“教 agent 怎么做某类事”

**为什么你需要关心**

- 因为重复解释工作方法很浪费
- `skill` 适合沉淀流程、检查清单、领域知识和交付模板

**初学者第一次通常怎么用**

- 把高频任务整理成 `SKILL.md`
- 例如“测试补全规范”“代码评审清单”“文档撰写模板”

**什么时候先不要做**

- 如果你要的是访问外部系统，不要先想 `skill`
- `skill` 擅长的是方法复用，不是外部工具接入

**正例**

- 让 agent 按团队固定清单做代码评审
- 让 agent 按固定结构输出需求分析文档

**边界例**

- 如果你需要访问 Jira，不该用 `skill` 解决，而该看 `MCP`

### 3.9 项目专用工具：`custom tools`

**它是什么**

- 官方文档把 `custom tools` 定义为你自己创建、可在对话中被 LLM 调用的工具函数
- 它们和内置工具并列工作，可以定义在 `.opencode/tools/` 或全局工具目录中

**为什么你需要关心**

- 因为很多时候你需要的不是“更复杂的运行时扩展”，而只是“多一个项目专用动作”
- 这比直接写 `plugin` 更轻

**初学者第一次通常怎么用**

- 给项目加一个“读取本地配置并生成模块脚手架”的工具
- 给项目加一个“查询本地测试数据状态”的工具

**什么时候先不要做**

- 如果你只是想让 agent 按固定流程做事，先用 `skill`
- 如果你要接的是已有外部服务生态，先看 `MCP`

**正例**

- 给项目加一个“读取本地配置并生成模块脚手架”的工具

**边界例**

- 如果你要接的是标准外部服务生态，不优先自己造 `custom tools`，先看 `MCP`

### 3.10 外部工具接入：`MCP`

**它是什么**

- `MCP` 是 `Model Context Protocol`
- 官方文档把它定义为一种把外部工具和服务接入 OpenCode 的方式，支持本地和远程 MCP server

**为什么你需要关心**

- 因为这是把 GitHub、Jira、数据库、浏览器自动化、Sentry 之类外部能力带进 OpenCode 的主流方式

**初学者第一次通常怎么用**

- 当你想让 AI 读 GitHub issue、查 Jira、访问 Postgres、调用浏览器工具时，优先考虑 `MCP`
- 配置完成后，MCP 工具会与内置工具一起可用

**什么时候先不要做**

- 如果你只是想让 agent 按固定流程工作，不要先上 `MCP`
- 如果你还没想清楚哪些外部工具真的需要，不要一次启用很多 MCP server

**正例**

- 连接 GitHub、Jira、Postgres、浏览器自动化

**边界例**

- 如果你只是想让 agent 按团队评审模板工作，不该先上 `MCP`，而该用 `skill`

**额外提醒**

- 官方文档明确提醒：MCP server 会占用上下文空间
- 某些 MCP server，尤其像 GitHub 这类，可能明显增加 Token 消耗
- 因此初学者应从最少的 MCP 开始，按需启用

### 3.11 运行时扩展：`plugin`

**它是什么**

- 官方文档把 `plugin` 定义为“通过挂钩各种事件和自定义行为来扩展 OpenCode”的机制
- 插件可以从本地目录加载，也可以从 npm 包加载

**为什么你需要关心**

- 因为它代表 OpenCode 的高级扩展能力
- 但这不是大多数初学者第一周就需要碰的东西

**初学者第一次通常怎么用**

- 大多数情况下，先了解它的定位就够了
- 如果项目已经有现成插件，可以先学会使用，而不是立刻自己开发

**什么时候先不要做**

- 如果你只是想给 AI 增加一个外部工具，不要先写 `plugin`
- 如果你只是想复用某种方法，也不要先写 `plugin`

**正例**

- 监听 OpenCode 事件并同步到企业内部系统
- 扩展运行时行为，或在生命周期里注入组织级能力

**边界例**

- 如果只是想让 agent 多一个外部工具，不该写 `plugin`，而该优先看 `MCP` 或 `custom tools`

**一个容易忽略的事实**

- 插件也可以添加自定义工具
- 但从初学者角度，若你只是想补一个项目专用动作，优先用 `custom tools` 更简单
- 只有当你真的需要 hook 事件、组合多个扩展点或修改默认运行时行为时，再考虑 `plugin`

### 3.12 到底该选哪个：一张表先记住

| 你的目标 | 优先选项 |
| --- | --- |
| 让 AI 按固定流程分析、评审、测试 | `skill` |
| 给项目补一个专用工具动作 | `custom tools` |
| 让 AI 使用 GitHub、Jira、数据库、浏览器等外部能力 | `MCP` |
| 扩展 OpenCode 本身的事件或运行时行为 | `plugin` |

如果你只记一句话，可以记这个：

**先解决工作方法，再补工具能力，最后才扩展运行时。**

## 4. 真正好用的方式，是让它按工程流程工作

如果你只是让 AI “直接开始写”，你很快就会遇到偏题、返工和验证不完整的问题。更稳妥的方式，是让它按一个真正的工程流程走。

### 4.1 阶段一：需求分析

**这一阶段的目标**

- 把模糊需求变成明确边界

**推荐输入模板**

```text
请先不要写代码。
基于当前仓库和以下需求，完成需求分析：
[需求描述]

输出必须包含：
1. 需求目标
2. In Scope / Out of Scope
3. 可能受影响的文件
4. 技术风险和不确定点
5. 推荐的最小实现方案
6. 验收与验证清单
```

**推荐输出格式**

- 目标
- 范围
- 风险
- 方案
- 验证项

### 4.2 阶段二：方案设计

**这一阶段的目标**

- 把分析结果收敛成可执行步骤

**推荐输入模板**

```text
请基于前面的分析，把实现方案收敛成最小改动计划。
要求：
1. 优先复用现有结构
2. 明确每一步改哪些文件
3. 明确每一步如何验证
4. 如果有不确定点，先列出来，不要直接拍脑袋实现
```

**推荐输出格式**

- 第 1 步改什么
- 第 2 步改什么
- 每一步怎么验证
- 哪些点仍需确认

### 4.3 阶段三：开发实现

**这一阶段的目标**

- 按小步推进，降低错误成本

**推荐输入模板**

```text
现在开始实施，但请分步骤进行：
1. 先完成第一步改动
2. 展示关键改动点
3. 运行相关验证
4. 确认无误后再继续下一步
如果中途发现方案与现状不一致，请先停下来说明原因
```

**推荐输出格式**

- 本步修改内容
- 关键文件
- 变更原因
- 当前验证结果

### 4.4 阶段四：验证结果

**这一阶段的目标**

- 把验证当成正式阶段，而不是收尾动作

**推荐输入模板**

```text
请不要只告诉我“应该没问题”。
请实际运行能运行的验证命令，并按下面格式输出：
1. 运行了什么
2. 哪些通过
3. 哪些没跑
4. 剩余风险是什么
```

**推荐输出格式**

1. 运行了什么
2. 哪些通过
3. 哪些没跑
4. 剩余风险是什么

你真正需要的是“验证证据”，而不是“主观看起来没问题”。

### 4.5 阶段五：沉淀经验

一个需求做完后，建议至少做一项沉淀：

- 把项目约束补进 `AGENTS.md`
- 把高频方法整理成 `skill`
- 把项目专用动作整理成 `custom tools`
- 需要外部系统时再补 `MCP`
- 真有运行时集成诉求时再考虑 `plugin`

AI 开发效率真正会持续提升，不是因为你一次提示词写得很精彩，而是因为项目越来越“适合被 AI 理解和操作”。

## 5. 新手最容易掉进去的 6 个坑

### 坑一：把 `plugin` 当成新手入口

大多数初学者第一阶段根本不需要自己开发插件。先把 `session`、`permission`、`rules`、`skill` 用顺，再谈运行时扩展。

### 坑二：只研究 prompt，不管理 `context`

很多失败不是提示词不够聪明，而是上下文不够完整、不够准确、或者已经被无关信息污染。

### 坑三：把多个需求塞进一个 `session`

任务一混，历史消息、上下文压缩、分享和复盘都会一起变脏。

### 坑四：不理解 `Plan` / `Build` 和权限的关系

如果你只是把它们当成“两个不同语气的 agent”，那你还没有真正掌握 OpenCode 的使用方式。

### 坑五：一次启用太多 `MCP` server

官方文档已经明确提醒这会吃掉上下文。MCP 不是越多越好，而是越精确越好。

### 坑六：AI 改完代码却没有验证证据

“应该可以”“看起来没问题”都不算交付。你要看的是：跑了什么、通过了什么、没跑什么、剩余风险是什么。

## 6. 一个更省力的学习顺序

如果你准备认真把 OpenCode 用起来，我建议按这个顺序学习：

1. 安装 OpenCode 并接通 provider
2. 在项目里运行 `/init`，建立 `AGENTS.md`
3. 学会一个需求对应一个 `session`
4. 学会用 `Plan` 做分析，用 `Build` 做实施
5. 学会主动管理 `permission`
6. 学会补足 `context`
7. 针对重复任务沉淀 `skill`
8. 需要工具扩展时再学习 `custom tools`
9. 需要外部系统能力时再接 `MCP`
10. 只有当你要扩展 OpenCode 自身运行时，才去研究 `plugin`

从这个顺序看，你会发现：

- `skill` 通常比 `plugin` 更早有价值
- `MCP` 通常比“自己开发插件”更常见
- `permission` 比很多人想象得更重要

## 7. 最后两件事：隐私提醒和延伸阅读

### 7.1 分享与隐私提醒

官网首页强调，OpenCode 默认不存储你的代码或上下文数据；但官方 `Share` 文档也明确说明，一旦你启用了分享功能：

- 共享对话对任何拿到链接的人都是公开可访问的
- 对话历史会同步到服务器
- 在你明确 `/unshare` 之前，分享内容会持续可访问

因此，初学者至少记住三条：

- 不要分享包含密钥、敏感日志、专有代码的对话
- 分享前先检查内容
- 对敏感项目，可以直接在 `opencode.json` 中把 `share` 设为 `"disabled"`

## 最后一段总结

如果你只想记住一句话，可以记这个：

**OpenCode 的核心不是“会写代码”，而是“让 AI 在正确的上下文、权限和流程里参与开发”。**

真正适合初学者的路径也不是“先学插件开发”，而是：

`安装 -> /connect -> /init -> session -> Plan/Build -> permission -> context -> skill -> custom tools / MCP -> plugin`
